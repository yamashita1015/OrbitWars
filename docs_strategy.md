# 現在の戦略：Producer Hybrid（ETA-aware + Multi-size）

## 概要

slawekbielの公開コード「Producer Hybrid v4」をベースに、以下2つの改善を加えたルールベースエージェント。
機械学習は使用していない。PyTorchはテンソル演算のみに使用。

---

## 基本戦略：Flow Diff スコアリング

### 攻撃候補の生成
1. 自分の惑星（ソース）から敵・中立惑星（ターゲット）への全ペアを列挙
2. ターゲットは**距離が近い順**に最大12個（1v1）/ 7個（FFA）を候補に絞る
3. 防衛対象（18ターン以内に占領されそうな自惑星）を最大4個追加

### スコアリング（Flow Diff）
- 各候補について「この艦隊を送った場合、18ターン後の生産量ネット差分」を計算
- スコア = 自分の純増分 − 相手の純増分（競争的差分）
- **ROI閾値（1.5）を超えた攻撃のみ実行**

### Greedy選択
- スコア上位の波を最大6波（1v1）/ 6波（FFA）まで選択
- 1ターゲットにつき1波まで、ソース予算内で実行

## 改善1：ETA-aware Reinforcement Risk（beta=2.2）

### 問題
長距離攻撃は飛行中に敵が増援を送れるため、到着時点での守備数が増えているリスクがある。
元の実装はこれを無視しており、「一見とれそうな惑星」に無駄な艦隊を送ってしまっていた。

### 実装
攻撃に必要な最低艦隊数（capture_floor）に「敵の増援リスク」を上乗せする。

```
追加床 = β × ρ(ETA) × enemy_mass(target)

ρ(k) = clamp((k - eta_free) / eta_scale, 0, 1)
     = ETA3ターン以下: 0（敵に反応時間なし）
       ETA15ターン以上: 1（敵は確実に増援できる）
```

| パラメータ | 値 | 意味 |
|---|---|---|
| beta | 2.2 | 増援リスクの強さ |
| eta_free | 3.0 | リスクが始まるETA（ターン） |
| eta_scale | 12.0 | リスクが最大になるまでの期間 |

### 効果
- ローカルテスト：元バージョンに対して65-80%の勝率

---

## 改善2：Multi-size Candidates（0.5x + 1.0x）

### 問題
元の実装は毎回「safe_drain（最大艦隊数）」でしか攻撃候補を作らない。
最大を送ってしまうと、そのソース惑星から他のターゲットへ追加攻撃できなくなる。

### 実装
各（ソース, ターゲット）ペアに対して、2種類のサイズで候補を生成する：

| サイズ | 意味 |
|---|---|
| 0.5 × safe_drain | 半分送って残りを別ターゲットに使う |
| 1.0 × safe_drain | 全力送り（元の挙動） |

候補数が2倍になるが、Greedy選択により：
- T1に0.5xで攻撃 → 残り0.5xでT2にも攻撃 → 1ターンで2惑星占領
- T1に1.0xが必要な場合は引き続き全力送り

### 試したが効果がなかったもの
- 3分割（0.33, 0.5, 1.0）：計算コスト増、効果なし
- 3分割（0.5, 0.75, 1.0）：80ゲームで52.6%実効勝率、統計的ノイズ範囲内かつコスト増
- 生産力重みづきショートリスト：遠い高生産惑星を狙いすぎて逆効果
- 彗星への攻撃：盤外退出で艦隊ロスト → 逆効果
- 戦略整合（コサイン類似度）：有効方向バイアスが強すぎ、側面の良いターゲットを無視 → 逆効果
- 生産スノーボールブースト：遠い高生産惑星を優先しすぎ → 逆効果
- 動的horizon（勝ち/負けで伸縮）：勝っているとき12に縮んでプランニング精度低下 → 逆効果
- 動的beta（常に上昇）：等戦力でも2.86以上になり常に保守的すぎ → 逆効果
- `_suppress_late_candidates`（元の実装）：ターン380以降すべての中立攻撃にペナルティ → 逆効果

### 効果
- ローカルテスト（ETA-onlyとの比較）：60-73%の勝率

## 改善3：`_adjust_config`（動的ROI・波数調整）

### 問題
常に固定のROI閾値で動くと、劣勢時に攻撃が少なすぎて逆転できない。

### 実装
`run_turn` の先頭で毎ターン呼び出し。**劣勢時のみ**（ratio < 1.0）に以下を調整：

- **ROI閾値を最大0.30下げる**（`min_roi` まで）。deficit が大きいほど大幅に下げる
- **残りターン数 < 150 かつ ratio < 0.90 のとき**、時間切迫ボーナスを追加
- **波数を最大8まで増やす**（ratio < 0.70、または残り100ターン以内 + ratio < 0.95）
- 勝っている（ratio >= 1.0）ときは一切変更しない

### 注意
この機能を無効にすると勝率が約36%まで低下することが実証済み。必須機能。

## 改善4：Minimal Late-game Suppression（インライン）

### 問題
元の `_suppress_late_candidates` は終盤（ターン380以降）の中立攻撃すべてにペナルティを与えており、本来有効な攻撃機会も潰していた。

### 実装
`plan_lite_waves` の最後でインライン実装（lines ~486-491）。条件を絞り込んで副作用を最小化：

```
残り60ターン以内、かつ ETA >= remaining（到着が物理的に間に合わない）攻撃のみ -inf に設定
```

条件を二重に絞ることで、「終盤でも到着できる攻撃」は妨げない。

## 改善5：Shrinking Ring Conquest Multiplier

### 問題
スコアリングは各ターゲットを独立に評価するため、「コンパクトな前線を固める」という空間的な文脈がない。
遠い高スコアターゲットに手を伸ばして兵力が分散しやすかった。

### 実装
自陣惑星の艦隊数重み付き重心を中心に「征服リング」を定義する。

```
ring_radius = map_radius * (1 - t) + min_r * t   (t = step / 500)
```

- ゲーム序盤: リングが広い → ほぼ全ターゲットが inside
- ゲーム終盤: リングが縮小 → 重心付近だけ inside

各ターゲットのスコアにリング内外でそれぞれ乗数をかける：

| 位置 | 乗数 |
|---|---|
| ring 内（inside） | 1.3（30%ブースト） |
| ring 外（outside） | 0.7（30%ペナルティ） |

### パラメータ調整の経緯

| boost / penalty | Ring wins | Base wins | 判定 |
|---|---|---|---|
| 2.0 / 0.4（元ノートブック） | 18/40 (45%) | 20/40 (50%) | 逆効果 |
| 1.5 / 0.6 | 15/30 (50%) | 15/30 (50%) | 中立 |
| **1.3 / 0.7** | **46/80 (57.5%)** | **31/80 (38.75%)** | **採用** |

### 効果
- ローカルテスト（ring ON vs OFF, 80ゲーム合算）：57.5% vs 38.75%

## 改善6：Adaptive Distance Scale

### 問題
Flow Diff スコアは距離を考慮しない。遠い惑星は到達前に状況が変わるリスクが高いが、スコアが高ければ選ばれてしまう。

### 実装
`plan_lite_waves` 内、ring conquest の前に各候補のスコアに距離乗数をかける。

```
distance_scale = 1.0 / (1.0 + dist / ref)
ref = max(median_alive_planet_distance * 0.5, 1.0)
```

マップ上の生存惑星間距離の中央値を基準に正規化するため、マップサイズに依存しない。

### 効果
- ローカルテスト（ON vs OFF, 50ゲーム）：60% vs 38%（draws=1）

---

## 改善7：Proactive Defense (`_build_defense_entries`)

### 問題
Flow Diff の守備候補は「将来的に占領されそうな自惑星」を攻撃候補として加えるが、
増援を「積極的に送る」仕組みではなかった。

### 実装
`run_turn` の冒頭で `_build_defense_entries` を呼び、攻撃波と結合する。

- `defense_threat_horizon=14` ターン後に garrison が赤字になる自惑星を検出
- 最も近い自惑星ソースから 60% の艦隊を送る（最大3波/ターン）
- 攻撃エントリと concat してから `disambiguate_duplicate_launches` で整理

### 効果
- ローカルテスト（ON vs OFF, 50ゲーム）：50% vs 36%（draws=7）
- 引き分けが多い = 防衛が相手の突破を防いで膠着するケースが増える

---

## 改善8：Terminal Phase（終盤全力攻撃）

### 問題
通常の設定（ROI=1.5、波数6）はゲーム序盤〜中盤に最適化されている。
残り40ターンでは「ROI的に微妙な攻撃」でも実行すべきで、再集結に艦隊を使う余裕もない。

### 実装
`run_turn` で `_adjust_config` の直後に `_apply_phase_config` を呼ぶ。
残り40ターン（step >= 460）になったら**勝ち/負けに関係なく**無条件切り替え：

| パラメータ | 通常時 | 終盤（残り40ターン） |
|---|---|---|
| roi_threshold | 1.5 | **1.0** |
| max_waves_per_turn | 6 | **9** |
| enable_regroup | True | **False** |

`_adjust_config`（劣勢時のROI下げ）の後に適用するため、終盤は常にこの設定が優先される。

### 効果
- ローカルテスト（ON vs OFF, 50ゲーム）：56% vs 42%（draws=1）

---

### Committed-fleet penalty（逆効果・不採用）

50ゲームで 44% vs 50%。multi-size 送り (0.5x + 1.0x) と相性が悪い。
前ターンに 0.5x 送ったターゲットを翌ターンに追撃するのが正しい場面でもペナルティがかかる。
コードは残存するがデフォルト `committed_fleet_penalty=1.0`（無効）。

## Regroup（再集結）

攻撃に使わなかった余剰艦隊を、敵から圧力を受けている前線の自惑星に移動させる。

- 圧力 = 敵の近接艦隊数を距離減衰で加重したもの
- 最大7ターン以内に到達できる宛先へのみ移動
- 移動先が到着時点でもまだ自分のものである場合のみ実行

## 軌道予測（intercept_aim）

惑星は毎ターン軌道移動するため、直線を狙っても当たらない。
`intercept_angle` 関数がNewton-Raphson法で艦隊が命中する角度を計算する。
